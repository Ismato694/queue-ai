# Queue.ai — Architecture Audit (vs PRD & Design Docs)

**Date:** 2026-06-29
**Scope:** implemented code (`apps/web`, `services/worker`, `packages/shared`, `supabase/`) vs the planning docs (PRD `01`, API `05`, Architecture `06`, Flow Intelligence `07`, Security `08`, Founder Rules `09`, MVP Scope `09-MVP`).
**Method:** static read + grep verification of every claim (evidence cited).

> **Headline:** the product *works end-to-end and is well-shipped for a pilot*, but the **implemented architecture is a Supabase-direct (BaaS) system, not the NestJS API + modular-monolith specced in Phases 5–6** — and the **data/intelligence backbone (predictions, throughput, metrics) is declared but never populated**. The most business-critical logic (queue engine in SQL, RLS) has **no automated test coverage**, which is why four RLS/permission bugs only surfaced live. None of the findings are data-leaking in production today (RLS is enforced), but several are compliance- and scale-blocking before real patient data.

## Critical

### C1 — No automated coverage of the core logic; no DB-integration CI 🔴
**Evidence:** `npm test` covers `packages/shared` (state map), `assistant`, worker `messages` only. The **queue engine (`create_walkin_visit`, `call_next`, `complete_stage`, …)** and **RLS policies** live in SQL with no CI tests; `supabase/dev/smoke_test.sql` is manual. Four production-affecting bugs (`0017` RLS recursion, `0018` definer, `0020` schema grants) reached "done" and were only caught running against live Supabase.
**Deviation:** 14-TEST-PLAN lists DB/RPC + RLS as a level but it isn't automated; QA gate effectively bypassed for the riskiest code.
**Impact:** regressions in the state machine or a tenant-isolation policy would ship undetected — the worst class of bug for a multi-tenant health product.
**Fix:** add a CI job that spins ephemeral Postgres (or `supabase start`), applies `migrations/*`, runs `smoke_test.sql` + an RLS two-org isolation script as assertions. Gate merges on it. (Effort: ~1 day.)

### C2 — Patient phone stored in plaintext 🔴
**Evidence:** `0004_identity.sql` → `phone text unique not null` with a comment *"field-encrypt at app layer (08 §5)"* — never done.
**Deviation:** 08-SECURITY §5 promises field-level encryption for phone/emergency contact; NDPR data-protection posture.
**Impact:** a DB compromise or over-broad query exposes patient contact data; weakens the NDPR/DPA story before a real hospital.
**Fix:** encrypt `phone`/`emergency_contact` at rest (pgcrypto `pgp_sym_encrypt` with a KMS-held key, or app-layer envelope encryption) and search by a separate HMAC/blind-index column for the unique lookup. (Effort: ~1–2 days; do **before** real patient data.)

## High

### H1 — The intelligence/metric backbone is never populated 🟠
**Evidence:** grep confirms **no `insert into predictions | staff_throughput | daily_metrics | org_patterns`** anywhere. ETA/Flow Score/Hours Returned are computed **on read** (`get_visit_status`, `get_flow_overview`, `get_hours_returned`); the worker's `recomputeETA` was a stub and removed.
**Deviation:** 04-DATABASE §7 + 07-FLOW-INTELLIGENCE §10/§11 describe these as captured/written; the **Trust-Engine accuracy feedback loop (07 §11)** and **throughput "captured from day one" (04/07)** are unimplemented. F2/F13/F14 therefore run on live snapshots, not history.
**Impact:** (a) confidence can never self-calibrate against actuals → the honesty claim is unbacked long-term; (b) when the pilot ends there's **no throughput history** to train v2 — the "compounding data moat" doesn't compound (raw `activity_events` *are* written, so it's backfillable, but no consumer exists). 
**Fix:** worker jobs — (1) on each `call_next`/`complete`, log a `prediction` and later score it; (2) nightly roll `activity_events` → `staff_throughput` and `daily_metrics` (also makes H1/M5 go away); (3) `org_patterns` learner post-pilot. (Effort: 2–3 days.)

### H2 — Implemented architecture ≠ documented architecture 🟠
**Evidence:** there is no `services/api` / NestJS; the web client calls Supabase **PostgREST + RPC** directly; "real-time" is Supabase Realtime (no Redis, no Socket.IO, no `seq`). 
**Deviation:** 05-API (REST `/v1/...` contract, WebSocket `seq` channels) and 06-ARCHITECTURE (modular-monolith NestJS + workers + Redis) are **not what was built**. Phase 9 *locked* Supabase, which explains it — but 05/06 were never reconciled, so the canonical design docs now mislead any new engineer.
**Impact:** onboarding/governance risk; the "module seams = event bus" decoupling story in 06 doesn't exist on the client (see M3).
**Fix:** either (a) update 05/06 to document the actual BaaS architecture (recommended — cheapest, honest), or (b) reintroduce a thin API/edge layer if you later need server-side orchestration. Add an ADR recording the pivot. (Effort: ~half day for docs.)

### H3 — Anonymous `join_queue` has no abuse control 🟠
**Evidence:** `join_queue` is `SECURITY DEFINER`, granted to `anon`, callable with only a public `qr_token`; nothing rate-limits it.
**Deviation:** PRD anti-fraud + 08-SECURITY §12 (queue gaming / DoS). Pre-Queue protects *slot* occupancy but not *creation*.
**Impact:** a script can flood a branch with fake pre-queue visits (DB bloat, noisy displays, skewed metrics).
**Fix:** rate-limit per IP/phone (Supabase Edge Function gateway, a `pg` token-bucket on phone+branch, or Cloudflare in front); cap visits/phone/hour; CAPTCHA on the public join page. (Effort: ~1 day.)

### H4 — Staff status bypasses the event log 🟠
**Evidence:** `staff/page.tsx` → `sb.from('staff').update({ status }).eq('id', me.id)` (direct table write).
**Deviation:** 06 §4 invariant ("all mutations emit `activity_events`") and OPS-3 ("staff away → recompute ETAs + notify affected"). A break/return changes availability but writes no event, triggers no recompute, sends no "doctor delayed."
**Impact:** ETAs silently wrong when staff go on break; no audit of availability changes; the Predictive/Trust signals that read `staff.status` move without a trace.
**Fix:** `set_staff_status(p_status)` RPC that updates + logs an `activity_event` (+ enqueues notify). Route the staff toggle through it. (Effort: ~half day.)

## Medium

- **M1 Duplicated state machine.** `packages/shared` exports `STAGE_TRANSITIONS`/`canTransition`, but the SQL RPCs hard-code their allowed from-states (e.g. `state in ('serving','called')`). Two sources of truth → drift. **Fix:** a single `app.assert_transition(from,to)` in SQL as the authority; keep the TS map for the client and add a test asserting parity.
- **M2 Duplicated wait/ETA SQL.** The `ahead × avg ÷ servers` + `2400` baseline logic is re-implemented in `get_visit_status`, `get_flow_overview`, `get_hours_returned`. **Fix:** extract `app.stage_wait()` / `app.branch_baseline()` helpers; call from each.
- **M3 No data-access layer; `any`-typed rows.** 6 page files use `Record<string, any>` for DB data; RPC/table names are scattered through components. **Fix:** generate Supabase types (`supabase gen types typescript`) into `packages/shared`; wrap reads/writes in a small `lib/api.ts` so the UI doesn't import raw table names.
- **M4 Unscoped realtime.** All three dashboards subscribe to *every* `visit_stages` change and reload. RLS bounds *delivery* to the user's org (no leak), but it's coarse and re-fetches on unrelated rows. **Fix:** server-side filter by `branch_id`; apply the row delta instead of full reload; add the `seq`/reconcile from 05 §5.
- **M5 No metric rollup.** `get_hours_returned` lifetime scans all visits each dashboard load. **Fix:** the nightly `daily_metrics` rollup (H1) → read aggregates, not raw history.
- **M6 Simulation isn't event-replay.** `simulate_branch` is a capacity ratio, not the back-tested event-log replay 07 §6 promises. **Fix:** post-pilot, replay `activity_events` under counterfactual params; until then label it clearly as a rough estimate (it does).
- **M7 `combined.sql` duplication.** A generated copy of all migrations that can drift. **Fix:** generate it in CI (don't hand-maintain) or document it as build-output only.
- **M8 Duplicated queue UI.** `reception` and `staff` re-implement queue rendering/RPC plumbing. **Fix:** extract a `useQueue(branch,dept)` hook + shared `<QueueList>`.

## Low
- **L1** PHI **reads** aren't audited (08 said audit PHI access); only overrides/writes are. Add read-audit on patient-detail surfaces if compliance requires.
- **L2** `app.current_org()` runs a `staff` subquery on every RLS check (stable, cached per-query, but a cost at scale). Consider caching org in a JWT claim via a Supabase auth hook.
- **L3** Notification provider send is a console stub — expected pre-keys, but flag so it's not mistaken for delivery.
- **L4** No pagination on list reads (structure, queue) — fine at pilot volume.
- **L5** General doc drift (05/06/07 partially ahead of code); keep the decision log as the source of truth and reconcile docs.

## Recommended remediation order
**Before real patient data (pilot blockers):** C2 (encrypt phone) · C1 (DB-integration CI + RLS isolation test) · H3 (rate-limit join) · H4 (staff-status RPC).
**To make the product's promises true:** H1 (worker rollups → predictions/throughput/metrics) · M5 (rollup) · H2 (reconcile docs/ADR).
**Quality/scale hardening:** M1–M4, M6–M8.
**Opportunistic:** L1–L5.

**Quick wins (≤ half day each):** H4, M7, M4-filter, H2-docs. **Highest ROI:** C1 (prevents the class of bug you already hit 4×) and H1 (turns the data moat from claimed to real).

> Overall: a genuinely shippable pilot MVP with disciplined scope — but treat **C1, C2, H3, H4** as gating items before a hospital handles live patient data, and **H1** as the difference between "demo" and "compounding product."
